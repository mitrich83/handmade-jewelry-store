# Tooling & Code Quality

> Issue #5 · W1 Foundation · Реализовано в `feature/issue-5-eslint-prettier`

Этот документ описывает все инструменты обеспечения качества кода, настроенные в Issue #5.

---

## Почему конфиг-файлы в `.mjs`, а не `.ts`

Node.js не умеет запускать `.ts` нативно — TypeScript нужно сначала скомпилировать.
Инструменты (ESLint, lint-staged) читают свои конфиги напрямую через Node.js во время работы —
компиляции нет и быть не может. Поэтому: **исходный код приложения → `.ts`, конфиги инструментов → `.mjs` или `.json`**.

`.mjs` явно помечает файл как ES Module (можно `import/export`) без зависимости от `"type": "module"` в `package.json`.

## Файлы Issue #5 — назначение и ценность

| Файл | Тип | Что делает | Ценность для проекта |
|------|-----|------------|----------------------|
| `.prettierrc` | JSON конфиг | Задаёт правила форматирования: одинарные кавычки, без точки с запятой, 100 символов в строке, LF окончания строк | **Единый стиль кода.** Разные девелоперы (или разные сессии с Claude) не будут менять форматирование туда-обратно. Diff в PR будет содержать только смысловые изменения, а не пробелы |
| `.prettierignore` | Текст | Говорит Prettier какие файлы не трогать: `node_modules`, `.next`, `dist`, `pnpm-lock.yaml`, бинарные доки | **Скорость и безопасность.** Без этого Prettier пытался бы форматировать сгенерированные файлы и ломал бы их |
| `eslint.config.mjs` | ES Module JS | ESLint 9 flat config. Включает TypeScript ESLint (recommended), отключает конфликты с Prettier, добавляет правила из CODE_RULES: `no-any`, `no-unused-vars`, `consistent-type-imports` | **Автоматическая проверка правил кодирования.** Вместо ручного code review на `any` и неиспользуемые переменные — ESLint блокирует их на уровне коммита. Flat config — это ESLint 9, современный стандарт вместо устаревшего `.eslintrc` |
| `.lintstagedrc.mjs` | ES Module JS | Конфиг lint-staged: на `.ts/.tsx` файлах запускает `eslint --fix` + `prettier --write`, на остальных только `prettier --write` | **Скорость pre-commit хука.** Без lint-staged ESLint проверял бы весь проект (~30 сек). lint-staged берёт только файлы из `git add` и запускает инструменты только на них (~1-2 сек). Это ключевой фактор — медленные хуки обходят через `--no-verify` |
| `.husky/pre-commit` | Shell script | Один вызов: `pnpm lint-staged`. Git запускает этот файл автоматически перед каждым `git commit` | **Enforcement без участия человека.** Хук нельзя забыть запустить — он запускается сам. Плохой код физически не попадает в историю git. Все разработчики получают хук автоматически после `pnpm install` через `prepare` скрипт |
| `pnpm-lock.yaml` | YAML (авто) | Lock-файл pnpm. Фиксирует точные версии всех 159 установленных пакетов (включая транзитивные зависимости) | **Воспроизводимые сборки.** Без lock-файла `pnpm install` на другой машине или в CI может установить другие минорные версии, что приводит к "у меня работает, в CI падает". Lock-файл гарантирует байт-в-байт одинаковое дерево зависимостей везде |
| `package.json` (обновлён) | JSON | Добавлены 5 новых devDependencies: `eslint`, `typescript-eslint`, `eslint-config-prettier`, `husky`, `lint-staged`. Новые скрипты: `lint:fix`, `format:check`, `prepare` | **Декларация инструментов.** `prepare` скрипт — точка входа: при `pnpm install` Husky инициализируется автоматически. `format:check` используется в CI для проверки без изменения файлов |

## Изменения существующих файлов (авто-форматирование Prettier)

После настройки Prettier, существующие файлы были приведены к единому стилю.
Это одноразовое действие — после него все новые файлы сразу пишутся по правилам.

| Файл | Что изменилось |
|------|----------------|
| `.github/workflows/auto-pr.yml` | Выравнивание отступов и кавычки |
| `CONTRIBUTING.md` | Нормализация пробелов |
| `docker-compose.yml` | Нормализация отступов |
| `packages/shared/src/index.ts` | Одинарные кавычки в комментариях |
| `pnpm-workspace.yaml` | Нормализация |
| `README.md` | Добавлен перенос строки в конце файла |

## Цепочка инструментов

```
git commit
    │
    ▼
.git/hooks/pre-commit          ← создан Husky при pnpm install
    │
    ▼
pnpm lint-staged               ← читает .lintstagedrc.mjs
    │
    ├─ на *.ts, *.tsx:
    │      eslint --fix         ← читает eslint.config.mjs
    │      prettier --write     ← читает .prettierrc
    │
    └─ на *.json, *.md, *.yml:
           prettier --write
    │
    ▼
Ошибок нет? → коммит создаётся ✓
Есть ошибки? → коммит отменён, показывает что именно сломано ✗
```

## Как обойти хуки (только в экстренных случаях)

```bash
git commit --no-verify -m "emergency: ..."
```

> **Важно:** `--no-verify` не отменяет CI. GitHub Actions запустит те же проверки при пуше.
> Если обошёл хук — следующим коммитом обязательно исправь нарушения.

## Настройка CI (Issue #9, W1)

В CI окружении хуки не нужны — линтер запускается отдельным шагом.
При установке зависимостей в CI используй:

```bash
HUSKY=0 pnpm install --frozen-lockfile
```

- `HUSKY=0` — Husky видит переменную и не инициализируется
- `--frozen-lockfile` — запрещает изменение `pnpm-lock.yaml`, гарантирует воспроизводимость
